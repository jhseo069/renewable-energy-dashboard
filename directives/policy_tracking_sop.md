## 목표 A: 유관 부처 보도자료 RSS 수집

### 대상 부처 선정 기준

**포함 원칙**
- 신재생에너지 사업개발(풍력·태양광·ESS·계통 등)의 인허가·정책·기술 기준에 직접 영향을 주는 부처
- `korea.kr` 정책브리핑 RSS를 통해 인증 없이 수집 가능한 부처만 포함

**제외 원칙**
- RSS 미제공 부처 (별도 API·로그인 필요)
- 신재생 사업개발과 간접 관련만 있는 부처 (예: 국방부 — 해상풍력 군 협의 필요하나 보도자료 직접성 낮음)

### 수집 대상 부처 (4개)

| 부처 | 관련성 | RSS 출처 |
|------|--------|----------|
| 산업통상자원부 | RPS, 계획입지, 에너지 기본계획, ESS 보급 | `korea.kr` |
| 기후에너지환경부 | 신재생에너지 보급, 탄소중립 로드맵, 온실가스 | `korea.kr` |
| 해양수산부 | 해상풍력 공유수면 점용·사용 허가, 어업인 협의 | `korea.kr` |
| 산림청 | 육상풍력 산지전용 허가, 산림 형질 변경 | `korea.kr` |

### RSS 공용 엔드포인트
```
https://www.korea.kr/etc/rss.do
```
- 전체 부처 통합 보도자료 RSS
- 인증 불필요 (공개 API)
- 부처명 + 에너지 키워드 이중 필터로 관련 기사만 추출

### 에너지 관련 필터 키워드
```python
["풍력", "태양광", "신재생", "재생에너지", "ESS", "에너지",
 "해상", "계통", "RPS", "REC", "수소", "발전", "전력",
 "탄소중립", "온실가스", "허가", "인허가", "산지전용"]
```

### 예외 처리 원칙
- RSS 응답 실패 또는 빈 entries → **Dummy 데이터 반환** (앱 중단 금지)
- 부처별 독립 처리 (한 부처 실패가 다른 부처에 영향 없어야 함)

## 목표 B: 국회 오픈 API 법안 수집 (해상풍력)

### 대상 API
- **명칭**: 국회 오픈 API (열린국회정보)
- **발급처**: https://open.assembly.go.kr/portal/main.do
- **인증**: API 키 필요 (무료 발급, 최대 1,000건/일)
- **현재 상태**: API 키 미보유 → **Mock 데이터로 선행 개발**

### 추적 대상 법안 키워드
```python
["해상풍력", "신재생에너지", "재생에너지", "집적화단지",
 "분산에너지", "탄소중립", "전기사업법"]
```

### API 전환 조건
`.env` 파일에 `ASSEMBLY_API_KEY=<발급받은 키>` 설정 시 자동으로 실서버 전환.
키 미설정 상태에서는 Mock 데이터가 자동 반환됨.

### 수집 필드
| 필드 | 설명 |
|------|------|
| `bill_id` | 법안 고유 ID |
| `title` | 법안명 |
| `proposer` | 발의자 (의원 또는 정부) |
| `committee` | 소관 위원회 |
| `status` | 처리 상태 (심사중/소위심사/본회의 부의/가결 등) |
| `propose_date` | 발의일 |
| `link` | 국회 의안정보시스템 링크 |

## 코드 설계 원칙 (Reflector 페르소나)

1. **직관적 변수명**: `rss_entries` (O) / `data` (X)
2. **한국어 주석**: '왜' 이렇게 짰는지 반드시 기술
3. **부처별 독립 try/except**: 하나 실패해도 나머지 계속
4. **Dummy 데이터 마킹**: `is_dummy: True` 필드로 UI에서 구분 표시
5. **에너지 키워드 필터**: 부처 보도자료 중 무관 행정 기사 제거
